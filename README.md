# Online Retail Product Analytics

Портфолио-проект по продуктовой аналитике интернет-магазина на данных **Online Retail II**.

Проект показывает полный аналитический путь: от EDA и очистки транзакций до продуктовых метрик, RFM/retention-анализа, дизайна A/B-теста и опубликованного BI-дашборда в Yandex DataLens.

## Дашборд

Опубликованный дашборд: [Yandex DataLens](https://datalens.yandex/6k37hs9ketrgq)

Дашборд построен на подготовленных CSV-агрегатах из ноутбуков. Глобальные селекторы в текущей версии не используются осознанно: данные уже агрегированы на разных уровнях, поэтому фильтры могли бы вводить в заблуждение, просто скрывая строки в отдельных чартах вместо полноценного пересчёта метрик.

## Что сделано

- Проведён EDA датасета Online Retail II.
- Очищены транзакционные данные: технические операции, нулевые цены, складские корректировки.
- Восстановлены бизнес-сущности: строка заказа, заказ, клиент, товар.
- Посчитаны ключевые продуктовые метрики: revenue, AOV, refund rate, active customers, repeat purchase rate, purchase frequency.
- Подготовлены агрегаты по месяцам, странам, товарам, клиентам и когортам.
- Построены RFM-сегменты и cohort retention.
- Спроектирован будущий A/B-тест для гипотезы о блоке рекомендаций в корзине.
- Для AOV как ratio-метрики применены линеаризация и CUPED.
- Проведены A/A-симуляция, расчёт MDE/sample size и синтетическая A/B-симуляция.
- Собран и опубликован DataLens-дашборд на 3 страницы.

## Ключевые результаты

- Gross Revenue: около **20.1M**.
- Net Revenue: около **19.4M**.
- Sales Orders: около **39.5K**.
- Active Customers: около **5.9K**.
- AOV: около **509**.
- Refund Rate: около **3.6%**.
- Repeat Purchase Rate: около **72%**.
- Основная выручка сконцентрирована в United Kingdom: около **86%** gross revenue.
- Наиболее ценный RFM-сегмент - `Champion`: около **1.3K** клиентов и **11.9M** gross revenue.

## A/B-тест

В исходных данных нет настоящего A/B-теста, поэтому эксперимент не притворяется историческим. Вместо этого исторические данные используются для дизайна будущего теста:

- гипотеза: блок рекомендаций в корзине увеличит средний чек;
- unit of randomization: клиент;
- primary metric: `AOV = GrossRevenue / SalesOrders`;
- основной статистический подход: линеаризация AOV + CUPED + Welch t-test;
- дополнительная проверка: bootstrap для сырой AOV;
- guardrail: Return Orders Rate.

На исторических данных CUPED снизил дисперсию линеаризованной AOV примерно на **73%**. В синтетической A/B-симуляции группе B был добавлен uplift к revenue на **25%**; наблюдаемый прирост AOV составил около **21%**. Основной тест показал статистически значимый результат (`p-value ≈ 0.00023`), bootstrap также дал доверительный интервал выше нуля.


Файлы `data/*.csv` и `data/**/*.csv` не добавляются в git: исходный датасет нужно скачать отдельно, а все промежуточные и финальные CSV воспроизводятся запуском ноутбуков. Папки внутри `data/` сохранены через `.gitkeep`, чтобы ноутбуки могли сразу писать результаты в ожидаемую структуру.

## Ноутбуки

| Ноутбук | Содержание |

| `01_eda.ipynb` | загрузка данных, EDA, очистка, обработка возвратов и технических операций |

## BI-дашборд

Дашборд состоит из трёх страниц:

1. **Обзор бизнеса** - KPI, выручка, заказы, клиенты, возвраты.
2. **Драйверы выручки** - страны, товары, RFM-сегменты, retention.
3. **Готовность A/B-теста** - дизайн эксперимента, MDE, sample size, синтетический A/B и guardrail.

## Как воспроизвести

1. Установить зависимости:

```bash
pip install -r requirements.txt
```

2. Поместить исходный файл Online Retail II в:

```text
data/online_retail_II.csv
```

3. Убедиться, что в проекте есть папки `data/aggregates/` и `data/ab/`. В репозитории они уже сохранены через `.gitkeep`.

4. Последовательно запустить ноутбуки:

```text
notebooks/01_eda.ipynb
notebooks/02_product_metrics.ipynb
notebooks/03_ab_test.ipynb
```

## Ограничения данных

- Нет сессий, просмотров и пользователей без покупки, поэтому нельзя честно считать conversion rate и воронку.
- Возвраты нельзя надёжно связать с исходными заказами: в данных нет `original_invoice_id`.
- Настоящего A/B-теста в Online Retail II нет, поэтому экспериментальная часть - это дизайн и симуляция будущего теста, а не анализ исторического эксперимента.
- Строки без `Customer ID` используются для общей выручки и заказов, но не используются в RFM, retention и customer-level метриках.
